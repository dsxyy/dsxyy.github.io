# 整理与发布学习笔记

把学习材料先变成可审核草稿，只有用户明确批准当前版本后才能发布。

## 判定动作

- 用户要求总结、整理、沉淀或生成草稿：执行“整理阶段”。
- 用户提出修改：修改草稿，清除此前批准状态，重新展示完整草稿。
- 用户明确表示“当前版本审核通过并发布”“确认这版并推送”等：执行“发布阶段”。
- 意图含糊：只整理，不执行任何 Git 写操作。

不得把“看起来可以”“帮我处理一下”“继续”等表述视为发布批准。批准必须同时指向当前草稿和发布动作。

## 整理阶段

1. 收集当前对话、用户指定文件、代码片段、实验结果和必要上下文。事实不足时，明确标注待确认项，不编造结论。
2. 围绕一个主题组织内容，优先采用以下结构：
   - 问题或目标
   - 核心机制
   - 实验或实现
   - 结论
   - 限制与后续问题
3. 保留能解释机制的最小代码、命令、数据流或内存布局；删除无关聊天内容和重复叙述。
4. 使用中文 Markdown。标题具体，命令和代码注明语言，外部事实保留来源链接。
5. 将草稿保存到仓库外的本地临时目录，例如 `${TMPDIR:-/tmp}/publish-learning-notes/<slug>.md`。不要在整理阶段修改博客仓库。
6. 向用户展示完整草稿、拟定文件名和拟放置目录，明确说明“尚未发布”。等待审核。

## 审核闸门

将用户批准绑定到展示过的当前完整草稿。满足以下任一条件时批准失效：

- 正文、标题、文件名、目录或导航文字发生变化；
- 用户继续补充材料并要求合并；
- 无法确定用户批准的是哪个版本；
- 会话中没有展示待发布的完整版本。

批准失效后必须重新展示完整草稿并等待新的明确批准。

## 发布阶段

发布前读取 [references/blog-format.md](references/blog-format.md)，并按其中规则定位仓库和生成文件。

1. 确认当前草稿已通过审核闸门；否则停止并请求审核。
2. 定位 `dsxyy/dsxyy.github.io` 工作树；找不到时克隆到用户允许的工作区。
3. 检查远端为 `https://github.com/dsxyy/dsxyy.github.io.git` 或等价 SSH 地址，当前分支为仓库默认发布分支。
4. 运行 `git status --short`。存在非本流程产生的改动时停止，向用户报告；不得覆盖、暂存或清理这些改动。
5. 在写文件前执行 `git pull --ff-only`。失败时停止，不要 rebase、merge 或强推。
6. 写入文章并只更新必要的 `index.md` 导航项。展示文章路径、标题、导航位置和完整 `git diff --check`、`git diff -- <article> index.md` 结果。
7. 若 diff 与批准版本不一致，不得提交；重新进入审核。
8. 仅暂存文章文件和 `index.md`：禁止使用 `git add .`、`git add -A` 或通配符。
9. 使用简洁提交信息 `docs: add <topic> learning note`，然后推送当前发布分支。禁止强推。
10. 报告提交哈希、远端分支和站点文章地址；若 GitHub Pages 尚未更新，明确说明部署可能存在延迟。

用户对当前版本作出的“审核通过并发布”已同时授权步骤 8 至 9，无需再次询问。但认证失败、分支保护、远端冲突或工作树污染时必须停止。

## 输出要求

- 整理阶段以草稿内容为主体，结尾只报告草稿路径和未发布状态。
- 发布阶段以提交结果为主体，不声称未经验证的线上渲染结果。
- 始终区分事实、推断和待确认项。
